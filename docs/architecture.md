# 智伴随行：智能体总分架构课堂教学管理系统技术文档

> 版本：v1.1 | 更新日期：2026-05-28 | 适用案例：智伴随行：智能体总分架构课堂教学管理系统

---

## 1. 系统总览

### 1.1 核心概念

本系统是一个 **基于 OpenClaw 框架的多智能体教学辅助平台**。平台层由 `main` 负责跨智能体路由，业务层由多个平级课程智能体负责各自课程和班级群组。当前公开模板以 `assistant-4 / 小兰` 为主要代码样例，但架构上小兰与小墨是平级课程智能体：

| 课程智能体 | Agent ID | 负责课程 | 对接群组 | 业务定位 |
| --- | --- | --- | --- | --- |
| 小兰 | `assistant-4` | 《建筑工程测量》 | 7 个学生群 | 课程任务发布、答题批改、实操图片评价、学情汇总 |
| 小墨 | `assistant-2` | 《地基与基础》 | 6 个学生群 | 同类教学任务管理，服务另一课程和班级群组 |

系统同时存在两层调度关系：

1. **OpenClaw 平台总控层：** `main` 是平台级主智能体，负责把教师/管理者的跨智能体指令路由给小兰、小墨两个平级课程智能体。
2. **课程智能体总机-分机层：** 每个课程智能体在自己的业务域内再分为教师 DM 会话中的“总机”和各学生群会话中的“分机”。小兰服务《建筑工程测量》7 个群，小墨服务《地基与基础》6 个群。

本文档以小兰公开模板为主要示例说明课程智能体内部机制，但这些机制可平行迁移到小墨。必须先区分 `main` 和“课程智能体总机”：`main` 是跨智能体调度者；课程智能体总机是某一课程智能体内部的教学任务管理入口。

课程智能体业务域采用 **总机-分机** 架构：

| 角色 | 定义 | 运行方式 |
|------|------|---------|
| **总机** | 某一课程智能体的核心控制实例 | 教师私聊会话，负责出题、发布任务、汇总、清理 |
| **分机** | 课程智能体在各学生群组中的会话实例 | 每个群一个独立会话，负责接收学生答题、批改、回复 |

**关键理解：** 同一门课程内的总机和分机是 **同一个课程智能体** 的不同会话实例，不是不同的 AI。以小兰为例，总机和 7 个群分机共享同一套工作空间、知识库和任务管理工具，通过 `active_test.json` 实现跨会话状态同步；小墨则在自己的工作区和 6 个群中维护另一套独立状态。

### 1.2 一句话架构

平台级路径：

> **教师/管理者向 `main` 下达跨智能体指令 → `main` 按明确点名或既有路由关系选择小兰/小墨 → 通过 `sessions_send` / `sessions_spawn` 下发执行协议 → 子智能体在各自工作区执行并返回结果。**

课程智能体业务路径：

> **教师向某课程智能体总机下达指令 → 总机生成任务并发布到本课程工作区的 `active_test.json` → 各群分机监控学生消息 → 匹配任务并自动批改 → 通过 `task_manager.py` 回写提交状态。**

### 2.2 `main` → 子智能体调度层

`main` 是平台级薄总控。它不直接承担出题、批改、文书撰写等具体业务，也不判断业务内容或审核产出质量。它只负责 **路由选择、协议下发、跨智能体维护**。

```
                         ┌────────────────────┐
                         │ 教师 / 管理者        │
                         └─────────┬──────────┘
                                   │ 私聊 default 飞书账号
                                   ▼
                         ┌────────────────────┐
                         │ main · 平台总控      │
                         │ /workspace          │
                         │                     │
                         │ • 路由选择           │
                         │ • 协议下发           │
                         │ • 跨智能体维护       │
                         └─────────┬──────────┘
                                   │ agentToAgent
              ┌────────────────────┴────────────────────┐
              │                                         │
              ▼                                         ▼
     ┌────────────────┐                       ┌────────────────┐
     │ assistant-2    │                       │ assistant-4    │
     │ 小墨           │                       │ 小兰           │
     │ 地基与基础     │                       │ 建筑工程测量   │
     │ 6 个学生群     │                       │ 7 个学生群     │
     └────────────────┘                       └───────┬────────┘
                                  │
                                  ▼
                         ┌────────────────────┐
                         │ 课程总机 / 分机体系 │
                         │ active_test.json   │
                         │ 独立课程工作区      │
                         └────────────────────┘
```

**调度边界：**

| 层级 | 智能体/会话 | 主要职责 | 不应承担 |
|------|-------------|----------|----------|
| 平台总控 | `main` | 路由选择、协议下发、跨智能体维护 | 不判断任务内容，不审核产出质量，不直接批改学生答案，不直接改小兰业务状态文件 |
| 业务执行 | `assistant-4 / 小兰` | 《建筑工程测量》课程任务发布、学生答题批改、实操图片评价、提交统计 | 不处理小墨 6 个群的课程状态文件 |
| 业务执行 | `assistant-2 / 小墨` | 《地基与基础》课程任务发布、学生答题批改、实操图片评价、提交统计 | 不处理小兰 7 个群的课程状态文件 |

**`main` 的三项功能解释：**

| 功能 | 含义 | 边界 |
|------|------|------|
| **路由选择** | 根据用户是否明确点名“小兰/小墨”、既有业务归属、飞书账号/会话绑定关系，选择把指令发给哪个子智能体。 | 不分析任务内容质量，不替子智能体决定具体业务做法。 |
| **协议下发** | 把已确定要交给某个子智能体的事项整理成可执行指令，说明目标对象、可用文件、禁止事项、返回格式和异常说明要求。 | 不替子智能体执行任务，不把协议写成详细业务处理过程。 |
| **跨智能体维护** | 维护 `main` 可调度清单、`agentToAgent` 允许列表、工作区入口、规则文件一致性和路由关系说明。 | 不维护子智能体业务数据本身，不直接修改小兰 `active_test.json`。 |
 
**典型路由规则：**

| 用户意图 | `main` 应路由到 | 说明 |
|----------|----------------|------|
| 《建筑工程测量》随堂测试、图片实操任务、学生答题批改、提交汇总 | `assistant-4 / 小兰` | 小兰负责该课程 7 个学生群 |
| 《地基与基础》随堂测试、图片实操任务、学生答题批改、提交汇总 | `assistant-2 / 小墨` | 小墨负责该课程 6 个学生群 |
| 跨智能体规则、路由清单、配置一致性检查 | `main` 处理 | `main` 只维护调度关系和协议，不审核业务质量 |

**调度协议最小字段：**

`main` 下发给子智能体的消息应包含：

1. `任务目标`：要完成什么。
2. `目标对象`：目标群、文件、学生、课程或业务范围。
3. `执行边界`：哪些文件可读写、哪些状态不能直接改。
4. `必要工具`：如 `task_manager.py`、`kb_tool.py`、`message`、`image`。
5. `返回要求`：需要返回哪些结果、文件路径、统计或异常。

### 2.3 课程智能体的飞书身份

## 3. 总机-分机架构详解

> 本章的“总机”指 **课程智能体在教师 DM 会话中的业务总机**，不是平台级 `main`。以下以小兰为例；小墨采用同样结构，只是课程、群组、知识库和状态文件位于自己的工作区。

### 3.1 架构示意

```
                        ┌─────────────┐
                        │  教师A(教师) │
                        └──────┬──────┘
                               │ 私聊 (DM)
                               ▼
                    ┌──────────────────────┐
                    │  课程智能体 · 总机     │
                    │  (以 assistant-4 为例)│

                    │  • 接收教师指令        │
                    │  • 知识库检索出题      │

                       │              │
               写入 active_test.json  │ 发送消息

            ┌──────────────────────────────────┐

            │  my_document/test-data/           │
```

### 3.2 总机职责

总机运行在 **教师私聊（DM）会话** 中，是某一课程智能体业务域内接受教师直接指令的入口。若教师先找 `main`，则由 `main` 按课程归属将教学任务分派到对应课程智能体的总机会话。

| 职责 | 触发方式 | 输出 |
|------|---------|------|
| **出题** | 教师私聊要求生成测试 | `tests/test_YYYYMMDD_HHMMSS.json` |
| **发布** | 教师指定目标群发布 | 更新 `active_test.json` + 群消息 |
| **汇总** | 教师要求查看提交情况 | 统计报告（已提交/未提交名单、得分） |
| **清理** | 教师要求清理过期任务 | 归档到 `archive/` + 从 `active_test.json` 移除 |
| **状态查询** | 教师询问当前活跃任务 | 列出所有未完成任务 |

**出题流程：**
```
教师指令 → kb_tool.py FTS5检索知识库 → 生成选择题JSON → 教师审核 → task_manager.py publish → 发群
```

### 3.3 分机职责

分机运行在 **群组会话** 中，是某一课程智能体在每个学生群里的会话实例。它们自动运行，无需手动启动。

| 职责 | 触发方式 | 输出 |
|------|---------|------|
| **接收答题** | 学生在群发送文字/图片 | 自动进入答题流程 |
| **批改** | 匹配到活跃任务 | 逐题批改 / image工具分析图片 |
| **保存答案** | 批改完成 | `answers/{task_id}_{student_id}.json` |
| **更新状态** | 通过 task_manager.py | 学生加入 `submitted_students` |
| **回复学生** | 批改完成 | 反馈批改结果 |

**分机答题流程：**
```
学生消息 → 读 active_test.json → 遍历ALL任务匹配 → 读任务文件 → 批改 → 保存答案 → 标记提交 → 回复
```

### 3.4 总机与分机的关系

| 维度 | 说明 |
|------|------|
| **身份** | 同一个课程智能体的不同会话实例（如 assistant-4 / 小兰） |
| **会话** | 不同会话（DM vs 群聊），各自独立的上下文窗口 |
| **工作空间** | 同一课程内共享本课程工作区，如 `/workspace-agents/assistant-4/` |
| **状态同步** | 通过 `active_test.json` 文件系统同步，不依赖内存 |
| **并发** | 最多 4 个并发会话（maxConcurrent: 4） |
| **与 main 的关系** | 课程智能体总机可接收 `main` 分派的任务；复杂辅助任务可由 `main` 或会话工具另建任务会话，分机一般不主动创建新会话 |

**⚠️ 核心设计原则：文件系统是唯一真相源**

总机和分机之间 **不通过内存/会话通信传递状态**，所有任务状态变更都写入 `active_test.json`。这保证了：
- 分机不需要知道总机发布了什么——读文件就知道
- 总机不需要知道分机处理了什么——读 answers 就知道
- 即使总机或分机重启，状态不会丢失

## 4. 会话路由机制

### 4.1 消息如何路由到正确的会话

```
学生/教师发送消息
    │
    ▼
Feishu Gateway 接收事件
    │
    ├── DM (私聊) → 路由到对应课程智能体的 DM 会话 → **总机**
    │
    └── 群聊 → 路由到对应课程智能体的群会话 → **分机**
              │
              ├── 第1群 (oc_b12207a6...) → 第1组分机
              ├── 第2群 (oc_2f5b006e...) → 第2组分机
              ├── ...
              └── 测试群 (oc_beb624b9...) → 测试群分机
```

### 4.2 会话标识

每个会话由 `sessionKey` 唯一标识，格式由 OpenClaw 内部管理。分机会话是 **持久化的**——每次学生在群里发消息，都会激活对应群的分机会话，并恢复其上下文。

### 4.3 `main` 到子智能体的路径

`main` 到小墨/小兰两个子智能体不依赖飞书群消息自然路由，而是通过 OpenClaw 的 `agentToAgent` 能力调用跨会话工具：

| 工具 | 用途 | 典型场景 |
|------|------|----------|
| `sessions_send` | 向已有子智能体会话发送指令并等待回复 | 小任务、状态查询、要求某课程智能体发布/汇总任务 |
| `sessions_spawn` | 创建一次性子任务会话 | 批量检查、较耗时分析、需要隔离上下文的任务 |
| `sessions_list` | 查看可用会话和会话状态 | 找到目标子智能体 DM 或群会话 |

**路径 1：教师先找 `main`，由 `main` 按课程归属分派教学任务**

```
教师/管理者
    │ 私聊 default 飞书账号
    ▼
main（平台总控）
    │ 按课程归属选择 assistant-4 或 assistant-2
    │ sessions_send / sessions_spawn
    ▼
课程智能体业务总机（DM会话）
    │ 生成任务、发布、汇总
    ▼
active_test.json + message 工具
    │
    ▼
对应课程各群分机处理学生消息
```

**路径 2：教师直接找课程智能体**

```
教师/管理者
    │ 私聊课程智能体飞书账号
    ▼
课程智能体业务总机（DM会话）
    │
    ▼
active_test.json + 各群分机
```

这条路径绕过 `main`，适合明确属于某一课程智能体业务域的教学任务；但如果涉及小兰和小墨协同、跨工作区治理或路由配置维护，应优先走 `main`。

**路径 3：`main` 调度另一课程智能体**

```
教师/管理者 → main
                  │
                  └── sessions_send → assistant-2（小墨）
```

具体目标会话通常按以下格式理解：

```text
agent:<agent_id>:feishu:direct:<user_open_id>
agent:<agent_id>:feishu:group:<chat_id>
```

其中：

- `direct` 会话适合向子智能体的业务总机下达管理任务。
- `group` 会话适合向指定群分机注入任务上下文或刷新提示。
- `main` 不应绕过子智能体直接修改其业务文件；需要状态变更时，应让对应子智能体按自己的工作流和脚本执行。

### 4.4 配置一致性要求

`main` 能否调度子智能体取决于两处配置必须一致：

```json
{
  "agents": {
    "list": [
      {
        "id": "main",
        "subagents": {
          "allowAgents": ["assistant-2", "assistant-4"]
        }
      }
    ]
  },
  "tools": {
    "agentToAgent": {
      "enabled": true,
      "allow": ["main", "assistant-2", "assistant-4"]
    }
  }
}
```

**配置原则：**

- `main.subagents.allowAgents` 决定 `main` 可见和可调度的子智能体清单。
- `tools.agentToAgent.allow` 决定跨智能体通信工具允许哪些 agent 参与通信。
- 两者不一致时，会出现 `main` 认为能派发、但实际会话工具不可达的问题。
- 每次新增、删除或改名子智能体，都要同步检查这两处配置。

## 5. 任务管理系统

### 5.1 active_test.json — 系统总线

`active_test.json` 是整个系统的 **核心数据总线**，总机和分机都通过它协调工作。

```
总机发布任务 ──写入──▶ active_test.json ◀──读取── 分机匹配任务
分机提交答案 ──写入──▶ active_test.json ◀──读取── 总机汇总结果
```

**数据结构：**
```json
{
  "active_tasks": [
    {
      "id": "test_20260527_145118",
      "file": "test_20260527_145118.json",
      "type": "text_answers",
      "published_at": "2026-05-27T14:52:01+08:00",
      "expires_at": "2026-05-28T14:52:01+08:00",
      "groups": ["oc_GROUP_07"],
      "submitted_students": ["ou_xxx"],
      "completed": false,
      "question_count": 2
    }
  ]
}
```

### 5.2 task_manager.py — 唯一写入入口

**⚠️ 铁律：禁止直接修改 `active_test.json`！所有状态变更必须通过 `task_manager.py`。**

| 命令 | 功能 | 使用者 |
|------|------|--------|
| `publish` | 发布新任务（清理过期+原子写入+文件锁） | 总机 |
| `submit` | 记录学生提交（加入 submitted_students） | 分机 |
| `summary` | 汇总任务（统计+归档） | 总机 |
| `clean` | 清理过期/完成任务 | 总机 |

**并发安全：** 所有操作通过 `flock` 文件锁 + `transactional_update()` 保证原子性。

### 5.3 任务类型

| 类型 | 消息触发 | 批改方式 | 文件位置 |
|------|---------|---------|---------|
| `text_answers` | 学生发送 A/B/C/D | 逐题对比正确答案 | `tests/` |
| `photo_submission` | 学生发送图片 | image 工具分析 + 量化评分 | `tasks/` |

### 5.4 任务文件结构

**选择题任务（tests/）：**
```json
{
  "id": "test_YYYYMMDD_HHMMSS",
  "title": "测试标题",
  "course": "课程名",
  "questions": [
    {
      "id": 1,
      "question": "题干",
      "options": {"A": "...", "B": "...", "C": "...", "D": "..."},
      "answer": "B",
      "knowledge_point": "知识点"
    }
  ]
}
```

**图片任务（tasks/）：**
```json
{
  "task_id": "task_YYYYMMDD_HHMMSS",
  "task_type": "photo_submission",
  "title": "任务标题",
  "description": "任务描述",
  "photo_count": 1,
  "grading_criteria": {
    "checklist": [
      {"id": "format", "name": "格式规范", "checks": ["..."], "weight": 20}
    ],
    "pass_rule": "总分 ≥ 70 为合格"
  }
}
```

## 6. 数据流详解

### 6.1 出题发布流程（总机）

```
1. 教师私聊总机："生成测试，内容：XXX，数量：N"
2. 总机 → kb_tool.py 检索知识库
3. 总机 → 生成 tests/test_YYYYMMDD_HHMMSS.json
4. 总机 → 展示题目预览，请教师审核
5. 教师 → "发布到第X组"
6. 总机 → task_manager.py publish --type text_answers --file xxx --target 第X组
7. task_manager.py → 清理过期任务 → 写入 active_test.json（持 flock）
8. 总机 → message 工具发送题目到群
9. ✅ 完成
```

### 6.2 学生答题流程（分机）

```
1. 学生在群发送消息（文字或图片）
2. 分机 → 读 active_test.json（必须重新读！不能用缓存！）
3. 分机 → 遍历 ALL active_tasks：
   ├── 匹配 chat_id ∈ task.groups
   ├── 匹配消息类型（文字=text_answers, 图片=photo_submission）
   ├── 学生 ∉ submitted_students
   └── 未过期
4. 无匹配 → 轻量回复或忽略
5. 有匹配 → 读对应任务文件
6. 批改：
   ├── text_answers → 逐题对比答案
   └── photo_submission → image 工具分析 + checklist 打分（限1次image调用）
7. 保存答案到 answers/
8. task_manager.py submit → 更新 submitted_students
9. 回复学生批改结果
10. ✅ 完成
```

### 6.3 汇总流程（总机）

```
1. 教师私聊："汇总第X组提交结果"
2. 总机 → 读 active_test.json
3. 总机 → 遍历 active_tasks，匹配目标群
4. 总机 → 读 answers/ 下对应文件
5. 总机 → 生成汇总报告（已提交名单、得分、未提交名单）
6. ✅ 回复教师
```

## 9. 关键设计原则与教训

### 9.1 铁律清单

| # | 规则 | 原因 |
|---|------|------|
| 1 | **分机收到消息必须重新读 active_test.json** | 文件可能在两条消息之间被修改，缓存数据过期 |
| 2 | **必须遍历 ALL active_tasks** | 同时存在多个任务时，只看第一个会漏匹配 |
| 3 | **执行顺序：批改→保存答案→标记提交→回复** | 先标记提交会导致跳过批改 |
| 4 | **禁止直接写 active_test.json** | 必须通过 task_manager.py，否则破坏原子性和文件锁 |
| 5 | **同张图片最多调 1 次 image 工具** | 反复调用会导致死循环 |
| 6 | **每张照片处理上限 3 个 tool call** | 防止超时和重复分析 |
| 7 | **image 失败时禁止编造评价** | 必须保存 pending_review 等待人工复核 |

### 9.2 历史教训

| 日期 | 事件 | 修复 |
|------|------|------|
| 2026-05-26 | 群会话先收到文字后收到图片，没有重新读文件，漏掉 photo_submission 任务 | 铁律：必须现在读，不能回忆 |
| 2026-05-26 | 分机先标记提交再批改，导致跳过批改、没保存答案 | 铁律：先批改→保存→再标记 |
| 2026-05-27 | image 工具返回数据有歧义，thinking 手动验算导致死循环 | 铁律：同图最多 1 次 image，验算一次就出结论 |
| 2026-05-27 | text_answers 批改后没保存 answers、没更新 submitted_students | 补全 text_answers 完整流程 |
| 2026-05-27 | 评分凭感觉，没有明确扣分逻辑 | 量化打分规则：按 check 条目均分权重 |

## 11. 扩展与维护

### 11.1 新增群组

1. 在 `memory/groups.md` 添加群信息（chat_id 必须准确）
2. 在 `memory/students.md` 添加学生名单
3. 总机发布任务时即可自动路由到新群

### 11.2 新增任务类型

1. 在 `task_file_schema.md` 定义文件格式
2. 在 `AGENTS.md` 添加匹配规则
3. 在 `task_manager.py` 确保 publish/submit/summary 支持

### 11.3 备份策略

- `active_test.json` → 每次 publish 前自动清理过期并归档
- `answers/` → 不删除，永久保存
- `archive/` → 按日期自动归档已完成/过期任务

## 12. 术语表

| 术语 | 定义 |
|------|------|
| **main** | OpenClaw 平台级主智能体，负责路由选择、协议下发和跨智能体维护 |
| **子智能体** | 本文档中指 `assistant-2 / 小墨` 和 `assistant-4 / 小兰` |

| **sessions_send** | 向已有智能体会话发送指令的跨会话工具 |
| **sessions_spawn** | 创建一次性子任务会话的跨会话工具 |
| **active_test.json** | 系统总线，所有活跃任务的状态中心 |
| **task_manager.py** | 唯一写入入口，保证原子性和并发安全 |
| **text_answers** | 选择题任务类型，学生发送字母作答 |
| **photo_submission** | 图片任务类型，学生发送照片作答 |
| **grading_criteria** | 评分标准，定义 checklist 和权重 |
| **kb_tool.py** | 知识库 FTS5 检索工具，出题时检索参考内容 |
| **flock** | 文件锁机制，防止并发写冲突 |
